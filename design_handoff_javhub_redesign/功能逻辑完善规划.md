# JavHub 功能 / 逻辑层完善规划（交 Claude CLI 执行）

> 这份不碰 UI 样式，只谈**交互闭环、状态一致性、信息架构、功能完整度**。
> 每条给出：现状 → 问题 → 方案 → 落点文件 → 验收标准。按 P0→P3 优先级排，建议从上往下做。
> 所有方案都**基于现有 API（`api/index.js`）已支持的能力**，不臆造后端。

---

## P0 · 真 Bug / 断掉的核心闭环（必修）

### P0-1 ★ Today 首页点海报跳错地方
- **现状**：`views/Today.vue` 的 `openVideo()`（约 326 行）执行
  `this.$router.push({ path: '/search', query: { q: code } })`，把用户甩进搜索页。
- **问题**：全站**其他所有视图**（`Search.vue` / `Favorites.vue` / `Subscription.vue` /
  `Actor.vue` / `DiscoveryDetail.vue` / `InventoryActor.vue`）点影片都调用全局
  `openVideoModal(video, routePath)`（来自 `utils/modalState.js`，在 `App.vue` 挂了全局
  `<VideoModal>`），弹出**详情弹窗**（拉番号、磁力、流地址、收藏）。唯独新做的门面页 Today
  行为不一致 —— 最常用的"点海报"在首页是断的，体验割裂。
- **方案**：让 Today 改用全局弹窗，与全站统一。
  ```js
  // Today.vue <script> 顶部
  import { openVideoModal } from '../utils/modalState.js'
  // methods.openVideo 整个替换为：
  openVideo(video) {
    if (!video) return
    // 保留"最近查看"记忆（见 P0-2）
    try {
      const safe = { ...video,
        content_id: video.content_id || video.dvd_id || video.code || video.id,
        display_code: video.display_code || video.dvd_id || video.content_id }
      localStorage.setItem(HERO_STORAGE_KEY, JSON.stringify(safe))
    } catch (_) {}
    openVideoModal(video, '/')   // 与其它视图一致：打开全局详情弹窗
  }
  ```
- **落点**：`views/Today.vue`
- **验收**：在首页点 hero、订阅更新里的任意海报 → 弹出 VideoModal（能看到磁力/收藏），
  URL 不再跳 `/search`；与在搜索页点卡片行为一致。

### P0-2 ★ "继续观看"是假的（progress 永远 0、按钮不播）
- **现状**：hero 标 "继续浏览 / 继续播放"，带进度条；但 `heroProgress` 初始化为 `0`、
  **全程没有任何地方给它赋值**（没有播放位置数据源），进度条 `v-if="heroProgress>0"` 永不显示；
  "继续播放" 按钮只是 `openVideo`，并不播放。hero 数据其实来自 localStorage 里"上次点开的那部"。
- **问题**：把"最近查看的一部"伪装成"继续观看 + 播放进度"，是**误导性 UI + 死代码**。
  项目当前并没有"记录播放进度"的能力（`getStreamUrl` 只给流地址，无回看位置回写）。
- **方案（诚实化，二选一）**：
  - **A（推荐，轻）**：把语义改成事实——`kicker` 文案 "继续浏览"→**"最近查看"**；
    去掉进度条与 `heroProgress` 相关全部死代码；主按钮文案 "继续播放"→**"查看详情"**，
    点击走 P0-1 的 `openVideo`。hero 没有 last-opened 记录时，回退到"最新入库第一部"（现有 `loadHero` 已实现该回退）。
  - **B（重，仅当你确实想要播放进度）**：在 VideoModal 播放时把
    `{content_id, position, updatedAt}` 写入 localStorage（如 `javhub_playback_pos`），
    Today 读它驱动真实进度条与"继续播放"。需要 VideoModal 内嵌播放器支持 timeupdate，工作量大。
- **落点**：`views/Today.vue`（+ 方案 B 还需 `components/VideoModal.vue`）
- **验收**：A——首页不再出现永不填充的进度条；文案与行为自洽（"最近查看"+"查看详情"）。

## P1 · 功能完整度（兑现重设计承诺的闭环）

### P1-1 候选确认应是一等页面，别埋在 legacy `/downloads`
- **现状**：路由 `/downloads` 加载的是旧 `views/Home.vue`，候选审核 UI 在
  `features/candidates/DownloadCandidatePanel.vue` 里、挂在那个老视图内。Today 的
  "待处理 → 去确认" 和 stat 卡的"候选"都 `router.push('/downloads')`，落到老首页。
- **问题**：候选确认是订阅自动化的核心动作（API 很全：`approve / reject / bulk/reject /
  bulk/restore / enrich-magnet / process / runs / retry-failed`），却没有独立入口，
  与重设计规划的"候选确认"独立屏不符；和"下载任务"挤在一个 legacy 页里，职责混淆。
- **方案**：
  1. 新增路由 `/candidates` → 新 `views/Candidates.vue`（或把 `DownloadCandidatePanel.vue`
     提升为独立页的主体），复用现有 candidates API 与组件逻辑，不重写后端调用。
  2. 状态分段：待确认 / 已通过 / 已拒绝 / 缺磁力（计数来自 `getDownloadCandidateSummary`）。
  3. 行内动作：✓通过(`approveDownloadCandidate`) / ✗拒绝(`rejectDownloadCandidate`) /
     补磁力(`enrichDownloadCandidateMagnet` 或 `updateDownloadCandidateMagnet`)；
     批量：`bulkRejectDownloadCandidates` / `processDownloadCandidates`。
  4. **缺磁力的候选，"通过"禁用**（无磁力无法入队，这是已知约束）。
  5. Today 的"去确认"、stat"候选"改指 `/candidates`。
- **落点**：`router/index.js`、新 `views/Candidates.vue`、`features/candidates/*`（复用）、`views/Today.vue`
- **验收**：`/candidates` 可独立访问；通过/拒绝后计数实时变；缺磁力行通过键禁用；Today 入口指向它。

### P1-2 从卡片 / 详情直接订阅演员（重设计的招牌动作，当前缺失）
- **现状**：`AppleVideoCard.vue` 只把演员名渲染在副标题文本里，**没有订阅按钮**；
  订阅只能去 `/subscription` 页操作。重设计稿的"卡片下方演员订阅条（绿=已订阅/灰=可订阅）"未落地。
- **问题**：信息（这部片里有谁）与动作（订阅她）割裂，得跳页、再搜一次，闭环长。
  而 API 完全支持就地订阅：`getSubscriptionStatus(actressId)` / `toggleSubscription(data)` /
  `addSubscription` / `searchActors`。
- **方案**（建议落在**详情弹窗**而非每张卡，避免列表过载）：
  1. `VideoModal.vue` 的"出演演员"区，每位演员名旁加订阅切换按钮：
     进入时按需查 `getSubscriptionStatus`，点按 `toggleSubscription`，乐观更新 + 失败回滚。
  2. 翻译名优先、罗马字辅助（沿用 `utils/actorDisplay.js` 的显示规则，别让用户猜日文）。
  3. （可选）在"私人策展/收藏"页的影片卡下挂同款订阅条——量大时再做。
- **落点**：`components/VideoModal.vue`（主）、可选 `views/Favorites.vue`
- **验收**：详情弹窗里能看到每位演员的订阅态并就地切换，切换后去 `/subscription` 能看到一致结果。

### P1-3 "下载任务"应是真页面，而非 legacy Home.vue
- **现状**：`/downloads` = `views/Home.vue`（老综合页，混着候选+下载+其它）。
- **问题**：P1-1 把候选拆走后，下载任务本身也该有干净的一等视图（队列/进行中/已完成/失败 +
  暂停继续/移除/重试）。API 现有：`getDownloads / createDownload / deleteDownload`，
  以及 `streamJobs`(SSE) / `getJobs` 可做实时进度。
- **方案**：新 `views/Downloads.vue` 承接 `/downloads`，用 `getDownloads` 渲染任务、
  `deleteDownload` 移除；进度可接 `streamJobs` 实时刷新（已有 SSE 基建）。老 `Home.vue`
  确认无其它职责后退役或保留为重定向。
- **落点**：`router/index.js`、新 `views/Downloads.vue`、退役 `views/Home.vue`
- **验收**：`/downloads` 是专注的下载队列；移除/重试可用；进度实时（若接 SSE）。

## P2 · 信息架构 / 命名一致性（减少认知摩擦）

### P2-1 "影库"语义自相矛盾
- **现状**：`/library` **重定向到** `/library-organize?tab=check`（维护工作台：查重/命名/孤立文件）；
  而真正"浏览片库"在 `/search`。即：菜单叫"影库"的地方其实是"整理"，"浏览"藏在"搜索"里。
- **问题**：重设计的心智是"影库 = 浏览海报墙"，与现路由冲突，用户找"我的片库"会迷路。
- **方案**（命名归位，三件事各有清晰入口）：
  - **浏览** → `/library`（指向 `Search.vue` 的浏览态，或给它一个"全部影片"默认无搜索词的入口），菜单名"影库/片库"。
  - **整理** → `/library-organize`，菜单名"片库整理"（维护工作台，维持现状）。
  - **搜索** → 作为影库内的检索，不单列为顶级心智（或保留但从属影库）。
  - 同步更新 `appNavigation.js` 的标签与图标分组，让"浏览/整理/下载"三类各就各位。
- **落点**：`router/index.js`、`appNavigation.js`、可能 `views/Search.vue`（支持空查询=浏览全部）
- **验收**：点"影库"进的是海报浏览，不是查重工作台；"整理"独立可达。

### P2-2 Today stat 卡跳转校正
- **现状**：stat "影库"→`/search`，"待处理/候选"→`/downloads`。
- **方案**：随 P1-1/P2-1 调整——"影库"→`/library`（浏览），"候选/待处理"→`/candidates`，
  "下载中"→`/downloads`，"订阅演员"→`/subscription`。保证每张卡点进去 = 它字面说的地方。
- **落点**：`views/Today.vue`（`statItems` 的 `to`）
- **验收**：四张卡的目的地与标题语义一一对应。

## P3 · 健壮性 / 一致性打磨（有余力再做）

### P3-1 错误态可恢复，而非静默空
- **现状**：Today 多个 loader `.catch` 后置空数组（静默）。整页失败时用户只看到空，不知道是
  "真没有"还是"加载失败"。
- **方案**：区分"空"与"错"——失败时该区块给一行可点"重试"（复用 `AppleErrorState.vue`），
  调用对应单个 `loadXxx()` 重拉，而不是整页 `loadAll`。
- **落点**：`views/Today.vue`（+ 已有 `AppleErrorState` / `AppleEmptyState` 组件）

### P3-2 hero 回退与"最近查看"去重
- **方案**：`loadHero` 的 localStorage 记录加时间戳与有效性校验（番号失效/被删时回退到最新入库），
  避免 hero 卡在一部已不存在的片上。
- **落点**：`views/Today.vue`、`HERO_STORAGE_KEY` 相关。

### P3-3 订阅/收藏状态全局一致（避免多页不同步）
- **现状**：收藏有 `utils/favoriteState.js`、订阅有 `utils/subscriptionState.js` 做状态。
- **方案**：P1-2 就地订阅后，确保走这些既有 store/util 广播变更，使
  Today / 详情 / 订阅页三处状态一致（一处订阅，别处即时反映）。
- **落点**：`utils/subscriptionState.js`、`utils/favoriteState.js` 的消费方。

## 建议执行顺序
1. **P0-1 / P0-2**（首页两处，改动小、收益最大、立刻消除割裂感与误导）
2. **P1-1 候选独立页** → **P1-3 下载独立页**（把 legacy Home.vue 拆干净）
3. **P2-1 / P2-2 IA 归位**（命名与跳转对齐新心智）
4. **P1-2 就地订阅**（招牌闭环）
5. **P3** 健壮性收尾

## 一句话给 Claude CLI 的总纲
> 不动视觉样式。按本文件 P0→P3 逐条改交互闭环与路由：先让 Today 点海报走全局
> `openVideoModal`（别跳 /search）、把假的"继续观看"诚实化；再把候选确认、下载任务从
> legacy `Home.vue` 拆成独立视图；最后归正"影库=浏览 / 整理 / 下载"的命名与跳转。
> 每条都有对应的现有 API，不要新增后端假设；改动 guard 测试时同步更新断言。

# P1-1 执行蓝图 · 候选确认独立页 `/candidates`

> 目标：把"候选确认"从 legacy `views/Home.vue` 里**提升为一等页面**，但**不重写后端调用、不复制逻辑**。
> 关键约束：候选的全部状态/方法目前钉在 `Home.vue`，且多个 guard 测试（`WorkflowIntegration.test.js`、
> `Home.test.js`）断言这些字符串**就在 Home.vue 源码里**。所以正确做法是 **抽成 composable，两边复用**，
> 再把相关 guard 断言改指 composable。**纯展示组件 `DownloadCandidatePanel.vue` / `CandidateRunPanel.vue` 不动。**

---

## 架构决策
```
features/candidates/
  ├─ DownloadCandidatePanel.vue   (不动 · 纯展示，props in / emits out)
  ├─ CandidateRunPanel.vue        (不动)
  └─ useDownloadCandidates.js     (★新建 · 把 Home.vue 的候选状态+方法整体搬来，Composition API)
views/
  ├─ Home.vue                     (改：候选相关 data/methods 删除，改用 useDownloadCandidates())
  └─ Candidates.vue               (★新建 · 一等页面，薄壳，消费同一个 composable)
router/index.js                   (改：新增 /candidates)
appNavigation.js                  (改：导航加"候选确认"入口)
```
> 为什么用 composable 而非 Pinia store：候选状态是页面级、带路由查询同步，composable 最贴合现有
> Options/Composition 混用风格，改动面最小。若项目已有 store 习惯也可用 store，原则一致：**单一真相源，两处复用**。

---

## 步骤 1 — 新建 `features/candidates/useDownloadCandidates.js`
把 `Home.vue` 里**候选相关**的 state、computed、methods 整体迁入，导出一个 `useDownloadCandidates(options)`。
`options` 用来吸收两个页面的差异（路由前缀、是否同步 query、默认来源过滤）。

**要迁入的 state**（来自 Home.vue `data()` 约 171–172 行）：
```
candidates, candidateStats, candidatePage, candidatePageSize(=50), candidateTotal, candidateTotalPages,
selectingCandidates, selectedCandidateIds, bulkCandidateLoading, candidateBatchProcessing,
candidateRuns, candidateRunsLoading, candidateMutations, magnetEditor,
candidateFilter { status, source, actress_id, q, needs_magnet, missing_cover, latest_event_action }
```

**要迁入的 computed**：
`candidateFilterLedger`、`candidateRepairScopeLabel`、`candidateEventActionLabel`、`processVisibleLabel`
（以及面板用到的 `candidateLatestEventFilters`、`candidateRepairScope` 等——读面板模板把它们补全）。

**要迁入的 methods**（Home.vue 约 242–350 行，逐个搬，**API 调用一字不改**）：
```
loadCandidates, loadCandidateSummary, loadCandidateRuns,
updateCandidateSearch, submitCandidateSearch,
approveCandidate, rejectCandidate, processCandidate, enrichCandidateMagnet,
bulkRejectCandidates, bulkRestoreCandidates, processVisibleCandidates, retryFailedCandidateRun,
openMagnetEditor, closeMagnetEditor, submitMagnetEditor,
isCandidateMutating, setCandidateMutation, clearCandidateMutation,
candidateFilterPayload, candidateRouteQuery, syncCandidateRoute(可参数化路由前缀)
```
> 用到的依赖：`api`、`requestConfirm`（`utils/confirmDialog.js`）、`homePresentation`、`cleanObject`、
> `$message`。迁入后改为显式 import / 注入，不再依赖 `this`。Composition 写法用 `ref/reactive/computed`，
> 路由用 `useRoute/useRouter`。返回一个对象，把上面所有 state/computed/方法暴露出去。

**`options` 形参建议**：
```js
useDownloadCandidates({
  routePath = '/candidates',   // 路由同步落到哪个 path
  syncRoute = true,            // 独立页 true；若 Home 想保留旧 query 行为，按需
  defaultSource = '',          // 例如未来给 Inventory 复用时传 'inventory'
} = {})
```

## 验收标准
1. `/candidates` 可独立访问，导航有入口，Today 的候选入口指向它。
2. 待确认/已通过/已拒绝/待补磁力分段切换 + 计数实时（来自 `getDownloadCandidateSummary`）。
3. 行内 ✓通过 / ✗拒绝 / 补磁力 / 处理 可用；批量拒绝/恢复/按策略处理可用；runs 重试可用。
4. **缺磁力候选的"批准/通过"禁用或给出明确提示**（沿用 `approveCandidate` 里 magnet 缺失的 details 文案）。
5. Home.vue 不再是候选的唯一真相源；`npm test` 全绿（含改过的 guard 与新增冒烟测试）。
6. 行为与原 Home 候选 tab 一致——这是"搬家"不是"重写"，功能不回退。

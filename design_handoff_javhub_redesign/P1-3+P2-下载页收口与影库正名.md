## P1-3 · `/downloads` 收口为纯下载任务页

### 现状
`views/Home.vue`（`/downloads`）现有 3 个 tab：**真实任务**(TaskList) / **下载候选**(DownloadCandidatePanel) / **下载源**(DownloaderManagementPanel)。
P1-1 之后，候选已有一等页面 `/candidates`（共用 `useDownloadCandidates` composable）。
→ Home 里的"下载候选"tab 成了**重复入口**：同一份可变数据有两个交互面，正是要消除的状态分叉风险。

### 改动
1. **删除候选 tab 及其面板**：移除 `Home.vue` 模板里
   - `<DownloadCandidatePanel v-if="activeTab === 'candidates'" .../>` 整块；
   - `.download-tabs` 里的"下载候选"按钮（含 `tab-badge`）；
   - 顶部副标题中 `candidateStats.candidate` 的提示 span；
   - `candidateFilterLedger` 那段 `v-if` 块。
2. **退掉 composable 接线**：`setup()` 里删除 `useDownloadCandidates(...)` 调用、相关解构、
   `DownloadCandidatePanel` 的 `defineAsyncComponent` import 与 `components` 注册、
   `openCandidateTab` / `candidateActive` 等只服务候选 tab 的逻辑。
   保留：tasks（TaskList）与 downloaders（DownloaderManagementPanel）两条线**原样不动**。
   `activeTab` 取值集合从 `['tasks','candidates','downloaders']` 收成 `['tasks','downloaders']`，默认 `tasks`。
3. **标题对齐导航**：页头 `<h1>下载管理</h1>` → `下载任务`（与 `appNavigation.js` 标签一致）。
4. **旧深链兼容**：`/downloads?tab=candidates` 不再有对应 tab。在 `router/index.js` 加一条
   `beforeEnter` 或在 Home `setup` 里：若 `route.query.tab === 'candidates'`，`router.replace('/candidates')`。
   （`Actor.vue` / `LibraryOrganize.vue` 里仍有指向 `/downloads?tab=candidates` 的跳转——一并改指 `/candidates`，
   grep `tab: 'candidates'` 与 `tab=candidates` 全量替换。）

### Guard 测试同步
- `views/Home.test.js`：删/迁断言——`DownloadCandidatePanel` 在 Home 的挂载、候选 props、`activeTab==='candidates'`
  相关（约 66/82/83/91/95/96/106/107/114/115）。候选行为已由 `Candidates.test.js` 覆盖，这里只保留 tasks/downloaders 断言。
- `views/WorkflowIntegration.test.js`：确认其候选断言已指向 `useDownloadCandidates.js`（P1-1 已迁）；若仍有
  读 `home` 源码断言候选方法的，改指 composable。
- `appNavigation` 相关测试：`/downloads` 仍存在，无需动。

### 验收
- `/downloads` 只剩 **下载任务 + 下载源** 两 tab，无候选 tab；标题"下载任务"。
- 候选管理只在 `/candidates`；从 Actor/库存等跳"处理候选"都落到 `/candidates`。
- `/downloads?tab=candidates` 自动跳 `/candidates`。
- `npm test` 全绿。

## P2 · 影库命名归位（让"浏览片库"有正名入口）

### 现状（IA 错位）
- 导航"日常使用"里：浏览片库被叫 **"影片检索"** → `/search`；没有任何叫"影库/片库"的浏览入口。
- "影库/library"这个词被维护台占用：`router` 里 `/library` **重定向到** `/library-organize?tab=check`（查重/命名/孤立文件）。
- 结果：用户想"看我的片库海报墙"，得点一个写着"检索"的入口；而点字面像"片库"的地方进的是查重工作台。心智拧着。

### 改动
1. **正名浏览入口**：`appNavigation.js` 把"日常使用"里
   `{ path: '/search', label: '影片检索', icon: IconSearch }` 的 **label 改为 `影库`**（或 `片库`）。
   底部导航 `bottomNavItems` 的 `{ path:'/search', label:'检索' }` → `影库`（保持一致，可选）。
   > 检索仍是该页内的功能（搜索框留着），只是入口正名为"影库"——参照 Apple Music"资料库"内含搜索。
2. **确保"影库"落地即浏览，而非空搜索框**：核对 `views/Search.vue`——
   若它**无查询词时默认展示全部影片**（已有则无需动）；若它需要先输入才出结果，则加一个默认浏览态：
   进入无 `q` 时调用 `api.listVideos(page, page_size)`（接口已存在）渲染海报墙，输入后再切检索。
   用现有 `AppleVideoCard` + poster-grid，**不新增样式系统**。
3. **`/library` 词义归位**：`router/index.js` 把
   `{ path: '/library', redirect: { path: '/library-organize', query: { tab: 'check' } } }`
   改为 `{ path: '/library', redirect: '/search' }`（"library"=浏览，不是整理）。
   同步 `appNavigation.js` 的 `navActivePaths`：把 `'/library'` 从 `'/library-organize'` 桶
   **移到** `'/search'` 桶：
   ```js
   '/search': ['/search', '/actor', '/library'],
   '/library-organize': ['/library-organize', '/inventory', '/duplicates', '/normalize'],
   ```
4. **"片库整理"保持不动**：`/library-organize` 标签"片库整理"语义正确（=维护），不改。

### Guard 测试同步
- 若有针对 `/library` 重定向到 `/library-organize` 的路由测试，改成断言重定向到 `/search`。
- `appNavigation` 测试里 `navActivePaths` 的 `/library` 归属断言，跟着改到 `/search` 桶。
- 导航标签若被测试断言为"影片检索"，改为"影库"。
- grep `影片检索`、`'/library'`、`tab: 'check'` 自查所有引用点。

### 验收
- 导航"影库"点进去 = 海报浏览（无查询词也有内容），搜索框仍可用。
- "片库整理"是独立的维护台，二者不再混淆。
- 直接访问 `/library` 落到浏览，不再落到查重工作台。
- `npm test` 全绿。
